[id="data-contracts-overview_{context}"]
== Overview

[role="_abstract"]
The Data Contracts framework enables teams to define, enforce, and track formal agreements about schema structure, ownership, and quality using the industry-standard ODCS format.

A data contract in {registry} combines:

* *ODCS format* - Contracts are submitted as ODCS v3.1 YAML documents and projected onto schema artifacts
* *Metadata* - Standardized ownership, classification, and support information stored as `contract.*` labels
* *Lifecycle* - Status tracking (DRAFT, STABLE, DEPRECATED) and promotion workflows
* *Rules* - Quality rules from ODCS contracts projected as CEL contract rules on schema artifacts
* *Field Tags* - Semantic annotation of schema fields (PII, SENSITIVE, EMAIL) stored as version labels, extracted from Avro, JSON Schema, and Protobuf schemas
* *Migration* - Schema evolution with data transformation between versions _(planned)_

NOTE: Data contracts is an experimental feature. You must set `apicurio.features.experimental.enabled=true` and `apicurio.contracts.enabled=true` to use contract endpoints.

[id="data-contracts-odcs_{context}"]
== ODCS native format

[role="_abstract"]
{registry} uses the Open Data Contract Standard (ODCS) v3.1 as its native contract format. ODCS is a Linux Foundation standard maintained by the Bitol project. When an ODCS contract is submitted, its contents are *projected* onto the referenced schema artifacts.

=== Why ODCS

* *Industry standard* - ODCS is the Linux Foundation standard for data contracts, with ecosystem tooling such as `datacontract-cli`
* *Vendor neutral* - {registry} is the first schema registry to natively support ODCS
* *Interoperable* - ODCS contracts can be imported/exported and used with external tools without translation

=== Projection model

When an ODCS contract is submitted to {registry}, the following projections are applied to the referenced schema artifact:

.ODCS projection mapping
[cols="2,2,2", options="header"]
|===
|ODCS section |Projected to |Target

|`info.status`
|`contract.status` label
|Schema artifact (draft->DRAFT, active->STABLE, deprecated->DEPRECATED)

|`info.dataClassification`
|`contract.classification` label
|Schema artifact

|`team.name`
|`contract.owner.team` label
|Schema artifact

|`team.domain`
|`contract.owner.domain` label
|Schema artifact

|`team.contact`
|`contract.support.contact` label
|Schema artifact

|`quality.accuracy` rules
|CEL contract rules (prefixed `odcs:`)
|Schema artifact contract_rules table

|`schemas[].fields[].pii`
|`field-tag.{field}|PII` version label
|Schema artifact latest version

|`schemas[].fields[].tags`
|`field-tag.{field}|{tag}` version labels
|Schema artifact latest version

|`serviceLevel.*`
|`contract.sla.*` labels
|Schema artifact (stored, not enforced)

|`quality.freshness`
|`contract.quality.freshness.*` labels
|Schema artifact (stored, not enforced)
|===

=== ODCS contract example

.ODCS v3.1 contract
[source,yaml]
----
apiVersion: v3.1.0
kind: DataContract
id: orders-contract
info:
  title: Orders Contract
  version: 1.0.0
  description: Data contract for order events
  status: active
  dataClassification: confidential
team:
  name: orders-team
  domain: commerce
  contact: orders@company.com
schemas:
  - name: OrderEvent
    type: avro
    location: orders/OrderEvent:3
    fields:
      customerEmail:
        description: Customer email address
        pii: true
        tags:
          - PII
          - EMAIL
      totalAmount:
        description: Total order amount in cents
quality:
  accuracy:
    - name: positive-amount
      expression: totalAmount > 0
      threshold: 1.0
    - name: valid-email
      expression: "customerEmail.matches('.*@.*\\\\..*')"
      threshold: 0.99
serviceLevel:
  availability: 0.999
  latency:
    p99: PT1S
----

When submitted, this contract:

1. Sets `contract.status=STABLE`, `contract.owner.team=orders-team`, `contract.classification=CONFIDENTIAL` labels on the `orders/OrderEvent` artifact
2. Creates two CEL contract rules (`odcs:positive-amount` with onFailure=ERROR, `odcs:valid-email` with onFailure=DLQ) on the artifact
3. Sets `field-tag.customerEmail|PII=EXTERNAL` and `field-tag.customerEmail|EMAIL=EXTERNAL` version labels
4. Stores SLA labels for external monitoring tools

=== Quality rule threshold mapping

ODCS accuracy rule thresholds are mapped to contract rule failure actions:

* *1.0 (100%)* -> `ERROR` - Zero tolerance, reject non-compliant data
* *0.95-0.99* -> `DLQ` - Near-100%, route violations to dead letter queue
* *Below 0.95* -> `NONE` - Soft constraint, log only

=== Manual rules are preserved

When an ODCS contract is updated, only rules prefixed with `odcs:` are replaced. Rules added manually through the contract ruleset API are preserved.

[id="data-contracts-metadata_{context}"]
== Contract metadata

[role="_abstract"]
Contract metadata provides standardized information about schema ownership, classification, and support. Metadata is stored using the reserved `contract.*` label namespace on artifacts.

=== Metadata fields

.Contract metadata fields
[cols="2,1,3", options="header"]
|===
|Field |Type |Description

|`contract.status`
|Enum
|Contract lifecycle status: `DRAFT`, `STABLE`, or `DEPRECATED`

|`contract.owner.team`
|String
|Name of the team responsible for the contract

|`contract.owner.domain`
|String
|Business domain (for example, payments, orders, users)

|`contract.support.contact`
|String
|Support email address

|`contract.classification`
|Enum
|Data classification: `PUBLIC`, `INTERNAL`, `CONFIDENTIAL`, or `RESTRICTED`

|`contract.stage`
|Enum
|Promotion stage: `DEV`, `STAGE`, or `PROD`

|`contract.lifecycle.stable-date`
|ISO-8601 date
|Date when the contract status became STABLE

|`contract.lifecycle.deprecated-date`
|ISO-8601 date
|Date when the contract status became DEPRECATED

|`contract.lifecycle.deprecation-reason`
|String
|Reason for deprecating the contract

|`contract.sla.availability`
|Number
|Target availability (for example, 0.999). Stored for external monitoring tools.

|`contract.sla.latency.p50`
|ISO-8601 duration
|Target p50 latency. Stored for external monitoring tools.

|`contract.sla.latency.p99`
|ISO-8601 duration
|Target p99 latency. Stored for external monitoring tools.

|`contract.odcs.contractId`
|String
|ID of the ODCS contract that projected this metadata

|`contract.odcs.contractVersion`
|String
|Version of the ODCS contract that projected this metadata
|===

=== Data classification levels

* *PUBLIC* - No restrictions, data can be freely shared
* *INTERNAL* - For internal use only, not for external parties
* *CONFIDENTIAL* - Need-to-know basis, limited access
* *RESTRICTED* - Highly sensitive data requiring strict access controls

[id="data-contracts-lifecycle_{context}"]
== Contract lifecycle

[role="_abstract"]
The contract lifecycle tracks the maturity and availability of a schema through defined status transitions.

=== Lifecycle statuses

* *DRAFT* - Initial state. The schema is being developed and is not ready for production use.
* *STABLE* - The schema is production-ready and consumers can rely on it.
* *DEPRECATED* - The schema is being phased out. Consumers should migrate to a newer version.

=== Status transitions

----
DRAFT ──────► STABLE ──────► DEPRECATED
   │                              ▲
   └──────────────────────────────┘
         (skip stable)
----

* `DRAFT` -> `STABLE` - Use when the schema is ready for production
* `STABLE` -> `DEPRECATED` - Use when phasing out the schema
* `DRAFT` -> `DEPRECATED` - Use to deprecate without ever reaching stable

NOTE: Reverse transitions (for example, `STABLE` -> `DRAFT` or `DEPRECATED` -> `STABLE`) are not allowed.

=== Promotion workflow (planned)

NOTE: Automated promotion workflow enforcement is planned for a future release. The promotion stage label can be set manually.

----
DEV ──────► STAGE ──────► PROD
----

[id="data-contracts-field-tags_{context}"]
== Field-level tags

[role="_abstract"]
Field tags provide semantic annotation of schema fields, enabling you to identify sensitive data and apply tag-based governance. Tags are stored as version-level labels using the convention `field-tag.{fieldPath}|{tagName}`.

=== Tag sources

Tags come from two sources:

* *Inline tags* - Extracted automatically from schema content during registration by tag extractors
* *External tags* - Projected from ODCS contracts or set manually via the version labels API

=== Supported schema formats

Tag extractors are available for the following schema formats:

==== Avro

Tags are extracted from `tags` and `confluent:tags` field properties:

[source,json]
----
{
  "type": "record",
  "name": "User",
  "fields": [{
    "name": "email",
    "type": "string",
    "tags": ["PII", "EMAIL"]
  }]
}
----

==== JSON Schema

Tags are extracted from `x-tags` and `x-confluent-tags` extension properties:

[source,json]
----
{
  "type": "object",
  "properties": {
    "email": {
      "type": "string",
      "x-tags": ["PII", "EMAIL"]
    }
  }
}
----

Supports nested objects, arrays, `additionalProperties`, and `allOf`/`oneOf`/`anyOf` compositions.

==== Protobuf

Tags are extracted from documentation comments using the `@tag:` annotation:

[source,protobuf]
----
message User {
  // @tag:PII,EMAIL
  string email = 1;
}
----

Also supports Confluent `confluent.field_meta` field options.

=== Tag label convention

Tags are stored as version-level labels with the following format:

----
field-tag.{fieldPath}|{tagName} = {source}
----

Examples:

* `field-tag.ssn|PII = INLINE` - PII tag extracted from schema
* `field-tag.customerEmail|EMAIL = EXTERNAL` - EMAIL tag projected from ODCS contract
* `field-tag.profile.address.zipCode|INTERNAL = INLINE` - Tag on nested field

=== Common tags

* `PII` - Personally Identifiable Information
* `SENSITIVE` - Sensitive data requiring special handling
* `EMAIL` - Email address fields
* `PHONE` - Phone number fields
* `FINANCIAL` - Financial data
* `HEALTH` - Health-related data (PHI)

[id="data-contracts-rules_{context}"]
== Contract rules

[role="_abstract"]
Contract rules enable you to define validation and transformation logic that can be associated with artifacts or specific versions. Rules are stored in the registry and organized into rulesets containing two categories: *domain rules* and *migration rules*.

When an ODCS contract is submitted, its `quality.accuracy` rules are automatically projected as CEL domain rules on the referenced schema artifact, prefixed with `odcs:`.

NOTE: Contract rules are currently stored and managed through the REST API. Runtime execution of rules in SerDes (serializers/deserializers) is planned for a future release.

=== Rulesets

A contract ruleset groups rules into two categories:

* *Domain rules* - Validation and transformation rules for business logic
* *Migration rules* - Rules for schema version migration

Rulesets can be defined at two levels:

* *Artifact-level* - Apply to all versions of an artifact
* *Version-level* - Apply to a specific version only

=== Rule kinds

* *CONDITION* - A validation rule that checks a condition and passes or fails
* *TRANSFORM* - A transformation rule that modifies data

=== Rule modes

* *WRITE* - Execute on serialize (producer-side)
* *READ* - Execute on deserialize (consumer-side)
* *WRITEREAD* - Execute on both serialize and deserialize
* *UPGRADE* - Execute during schema upgrade migration
* *DOWNGRADE* - Execute during schema downgrade migration

=== Rule actions

* *NONE* - Continue processing (log only)
* *ERROR* - Throw an exception and reject the operation
* *DLQ* - Route the message to a dead letter queue

=== Rule fields reference

.Contract rule fields
[cols="2,1,1,3", options="header"]
|===
|Field |Type |Required |Description

|`name`
|String
|Yes
|The rule name

|`kind`
|Enum
|Yes
|`CONDITION` or `TRANSFORM`

|`type`
|String
|Yes
|Rule executor type (for example, `CEL`, `CEL_FIELD`, `ENCRYPT`, `JSONATA`)

|`mode`
|Enum
|Yes
|When the rule is applied: `WRITE`, `READ`, `WRITEREAD`, `UPGRADE`, or `DOWNGRADE`

|`expr`
|String
|No
|The rule expression

|`params`
|Map
|No
|Key-value parameters for the rule executor

|`tags`
|List
|No
|Tags for categorizing or targeting the rule

|`onSuccess`
|Enum
|No
|Action on success: `NONE`, `ERROR`, or `DLQ`

|`onFailure`
|Enum
|No
|Action on failure: `NONE`, `ERROR`, or `DLQ`

|`disabled`
|Boolean
|No
|Whether the rule is disabled (default: `false`)
|===

== Schema migration rules (planned)

NOTE: Schema migration rules are planned for a future release and are not yet implemented.

[role="_abstract"]
Migration rules enable schema evolution with data transformation between versions.

=== Migration directions

* *UPGRADE* - Transform data from an older schema version to a newer one
* *DOWNGRADE* - Transform data from a newer schema version to an older one

== REST API

[role="_abstract"]
The Data Contracts REST API provides endpoints for managing contract metadata and contract rulesets. All endpoints require the data contracts feature to be enabled.

=== Contract metadata endpoints

.Contract metadata REST API endpoints
[cols="1,3,3", options="header"]
|===
|Method |Endpoint |Description

|GET
|`/groups/{groupId}/artifacts/{artifactId}/contract/metadata`
|Get contract metadata for an artifact

|PUT
|`/groups/{groupId}/artifacts/{artifactId}/contract/metadata`
|Create or update contract metadata for an artifact
|===

=== Example: Setting contract metadata

[source,bash]
----
curl -X PUT \
  http://localhost:8080/apis/registry/v3/groups/my-group/artifacts/my-artifact/contract/metadata \
  -H 'Content-Type: application/json' \
  -d '{
    "status": "DRAFT",
    "ownerTeam": "Platform Team",
    "ownerDomain": "payments",
    "supportContact": "platform@example.com",
    "classification": "CONFIDENTIAL",
    "stage": "DEV"
  }'
----

=== Example: Transitioning contract status

[source,bash]
----
curl -X POST \
  http://localhost:8080/apis/registry/v3/groups/my-group/artifacts/my-artifact/contract/status \
  -H 'Content-Type: application/json' \
  -d '{ "status": "STABLE" }'
----

[id="data-contracts-configuration_{context}"]
== Configuration

[role="_abstract"]
Configure the Data Contracts framework using {registry} application properties.

=== Enabling data contracts

.Required configuration properties
[cols="2,1,2", options="header"]
|===
|Property |Default |Description

|`apicurio.features.experimental.enabled`
|`false`
|Enable experimental features gate (required for data contracts)

|`apicurio.contracts.enabled`
|`false`
|Enable the data contracts feature
|===

Both properties must be set to `true` for data contracts endpoints to be accessible.

.Example: Enabling data contracts with environment variables
[source,bash]
----
export APICURIO_FEATURES_EXPERIMENTAL_ENABLED=true
export APICURIO_CONTRACTS_ENABLED=true
----
