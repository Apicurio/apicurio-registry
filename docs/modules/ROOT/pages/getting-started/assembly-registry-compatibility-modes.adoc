= {registry} schema compatibility modes

[role="_abstract"]

[id="registry-compatibility-overview_{context}"]
== Overview of compatibility rules

[role="_abstract"]
When you add a new version of an artifact to {registry}, you can configure the `COMPATIBILITY` rule to check whether the new content is compatible with existing versions.
Compatibility checking helps you ensure that producers and consumers can continue to work together without disruption when schemas evolve.

You can configure the `COMPATIBILITY` rule at three levels:

* *Global rules* apply to all artifacts in {registry} unless overridden.
* *Group-specific rules* apply to all artifacts in a group unless overridden by an artifact-specific rule.
* *Artifact-specific rules* apply to a single artifact and take the highest priority.

If you configure a rule at a higher level but want to disable it at a lower level, you must configure the same rule at the lower level and set its value to `NONE`.

Not all artifact types support compatibility checking.
For a list of supported artifact types, see the content rule maturity matrix in {registry-rule-reference}.

[id="registry-compatibility-mode-descriptions_{context}"]
== Compatibility mode descriptions

[role="_abstract"]
You can configure one of seven compatibility modes for the `COMPATIBILITY` rule type.
These modes determine which versions of the artifact {registry} uses for comparison and whether changes must be backward compatible, forward compatible, or both.

.{registry} compatibility modes
[%header,cols="1,3"]
|===
|Mode
|Description

|`NONE`
|All compatibility checks are disabled.
{registry} accepts any changes to the schema, regardless of whether they are backward or forward compatible.

|`BACKWARD`
|Clients using the new artifact can read data that the most recently added artifact wrote.
Use `BACKWARD` when you want to ensure that updated consumers can process existing data.

|`BACKWARD_TRANSITIVE`
|Clients using the new artifact can read data that all previously added artifacts wrote.
Unlike `BACKWARD`, which checks only against the latest version, `BACKWARD_TRANSITIVE` checks against every prior version.

|`FORWARD`
|Clients using the most recently added artifact can read data that the new artifact writes.
Use `FORWARD` when you want to ensure that existing consumers can process data that the updated schema produces.

|`FORWARD_TRANSITIVE`
|Clients using all previously added artifacts can read data that the new artifact writes.
Unlike `FORWARD`, which checks only against the latest version, `FORWARD_TRANSITIVE` checks against every prior version.

|`FULL`
|The new artifact is both forward and backward compatible with the most recently added artifact.
Both old and new consumers can read data that either the old or new version of the schema produces.

|`FULL_TRANSITIVE`
|The new artifact is both forward and backward compatible with all previously added artifacts.
`FULL_TRANSITIVE` is the strictest compatibility mode and guarantees complete interoperability across all schema versions.
|===

NOTE: The `BACKWARD` and `FORWARD` modes compare the proposed schema only with the most recently added version.
The transitive modes (`BACKWARD_TRANSITIVE`, `FORWARD_TRANSITIVE`, and `FULL_TRANSITIVE`) compare the proposed schema against all existing versions, which provides stronger guarantees but restricts the changes you can make.


[id="registry-compatibility-avro_{context}"]
== Avro schema compatibility

[role="_abstract"]
{registry} uses the Apache Avro library to check compatibility for Avro schemas.
The tables in each section describe common Avro schema changes and whether they maintain or break backward and forward compatibility.

=== Backward compatible changes for Avro

You can make these changes to an Avro schema and maintain backward compatibility.
Consumers using the new schema can still read data that the previous schema wrote.

.Avro backward compatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a field with a default value
|The new consumer reads old data successfully because the default value fills in the missing field.

|Remove a field that has a default value
|The new consumer ignores the removed field because it did not require the field.

|Add a new type to a union
|The new consumer handles the existing union types and the new one. For example, you can change `"string"` to `["null", "string"]`.
|===

=== Backward incompatible changes for Avro

If you make these changes, you break backward compatibility.
Consumers using the new schema cannot read data that the previous schema wrote.

.Avro backward incompatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a field without a default value
|Old data does not contain the new field, and the new consumer has no default value to use.

|Remove a type from a union
|Old data might contain the removed type, which the new consumer can no longer parse.

|Rename a field without an alias
|The new consumer cannot match the field to its previous name.

|Change a field type
|The new consumer expects a different data type than what the old data contains. For example, changing `string` to `int` breaks compatibility.
|===

=== Forward compatible changes for Avro

You can make these changes to an Avro schema and maintain forward compatibility.
Consumers using the previous schema can still read data that the new schema writes.

.Avro forward compatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a field with or without a default value
|The old consumer ignores unknown fields in data that the new schema produces.

|Remove a field that has a default value
|The old consumer fills in the missing field by using the default value from its own schema.
|===

=== Transitive compatibility example for Avro

Consider an example that demonstrates how transitive compatibility differs from non-transitive compatibility:

* *Version 1*: `{fields: [name: string]}`
* *Version 2*: `{fields: [name: string, email: string (default: "")]}`
* *Version 3*: `{fields: [name: string, email: string]}` (default removed from `email`)

In the example, Version 3 is backward compatible with Version 2 because the `email` field exists in Version 2 data.
However, Version 3 is _not_ backward compatible with Version 1 because the `email` field is missing and there is no default value.
As a result, Version 3 passes the `BACKWARD` check but fails the `BACKWARD_TRANSITIVE` check.


[id="registry-compatibility-json-schema_{context}"]
== JSON Schema compatibility

[role="_abstract"]
{registry} uses a custom library to analyze differences between JSON Schema versions.
Because JSON Schema uses a constraint-based validation model, a backward compatible change typically makes the new schema more permissive so that it accepts all data that the previous schema accepted.

=== Backward compatible changes for JSON Schema

You can make these changes to a JSON Schema and maintain backward compatibility.
The new schema continues to validate all data that the previous schema validated.

.JSON Schema backward compatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a new optional property
|Old data without the property remains valid under the new schema.

|Remove a property from `required`
|The new schema relaxes the constraint, so old data with or without the property remains valid.

|Decrease `minProperties`, `minLength`, or `minItems`
|The new schema relaxes the minimum constraint. Old data that met the stricter minimum still passes validation.

|Increase `maxProperties`, `maxLength`, or `maxItems`
|The new schema relaxes the maximum constraint. Old data that met the stricter maximum still passes validation.

|Add a value to `enum`
|The new schema expands the set of accepted values. Existing valid values remain valid.

|Change `additionalProperties` from `false` to `true`
|The new schema becomes more permissive by accepting additional properties.

|Extend the `additionalProperties` schema
|Making the additional properties schema more permissive accepts more data.

|Increase the size of `oneOf` or `anyOf`
|Adding alternatives to combined schemas accepts more data patterns.

|Decrease `minimum` or `exclusiveMinimum`
|The new schema relaxes the lower bound. Old data above the previous bound still passes validation.

|Increase `maximum` or `exclusiveMaximum`
|The new schema relaxes the upper bound. Old data below the previous bound still passes validation.

|Remove a `format` constraint
|The new schema removes a validation check, making it more permissive.

|Change `uniqueItems` from `true` to `false`
|The new schema relaxes the array constraint. Old data with unique items remains valid.
|===

=== Backward incompatible changes for JSON Schema

If you make these changes, you break backward compatibility.
The new schema rejects data that the previous schema validated.

.JSON Schema backward incompatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a property to `required`
|Old data without the newly required property fails validation.

|Increase `minProperties`, `minLength`, or `minItems`
|The new schema tightens the minimum constraint. Old data that met the previous minimum might fail validation.

|Decrease `maxProperties`, `maxLength`, or `maxItems`
|The new schema tightens the maximum constraint. Old data that met the previous maximum might fail validation.

|Remove a value from `enum`
|Old data that contains the removed value fails validation.

|Change `additionalProperties` from `true` to `false`
|Old data with additional properties fails validation.

|Change the `type` keyword
|Old data of the previous type fails validation. For example, changing `string` to `integer` breaks compatibility.

|Add a `pattern` or `format` constraint
|Old data that does not match the new pattern or format fails validation.

|Increase `minimum` or `exclusiveMinimum`
|The new schema tightens the lower bound. Old data below the new bound fails validation.

|Decrease `maximum` or `exclusiveMaximum`
|The new schema tightens the upper bound. Old data above the new bound fails validation.

|Decrease the size of `oneOf` or `anyOf`
|Removing alternatives from combined schemas causes some previously valid data to fail validation.

|Change `uniqueItems` from `false` to `true`
|Old data with duplicate items fails validation.

|Change a `const` value
|Old data with the previous constant value fails validation.
|===


[id="registry-compatibility-protobuf_{context}"]
== Protobuf schema compatibility

[role="_abstract"]
{registry} checks Protobuf schema compatibility based on the wire format contract.
Because Protobuf uses numeric field tags to identify fields on the wire, compatibility rules focus on preserving tag numbers and field types.

=== Backward compatible changes for Protobuf

You can make these changes to a Protobuf schema and maintain backward compatibility.
Consumers using the new schema can still read data that the previous schema wrote.

.Protobuf backward compatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a new field with a new tag number
|Old data does not contain the new tag, and the new consumer ignores the missing optional field.

|Add a new message type
|Old data is unaffected by the addition of new message definitions.

|Add a new RPC method to a service
|Old data and existing RPC calls are unaffected by the new method.
|===

=== Backward incompatible changes for Protobuf

If you make these changes, you break backward compatibility.
Consumers using the new schema cannot read data that the previous schema wrote.

.Protobuf backward incompatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Change a field tag number
|Old data uses the original tag number on the wire. The new consumer cannot match the field to its new tag.

|Change a field type
|The wire encoding differs between types. For example, changing `string` to `int64` causes parsing failures.

|Rename a field
|Although Protobuf uses tag numbers on the wire, renaming a field affects JSON serialization and generated code.

|Remove a field without reserving its tag
|Another field might reuse the tag number with a different type, which causes data corruption.

|Change a field label
|Changing between `optional`, `required`, or `repeated` alters the wire format and parsing behavior.

|Remove an RPC method from a service
|Clients that use the removed RPC method can no longer communicate with the service.

|Change an RPC method signature
|Changing the input or output message type breaks the RPC contract.
|===

=== Reserving fields in Protobuf

When you remove a field from a Protobuf schema, you should reserve both the tag number and the field name to prevent reuse:

----
message Person {
  reserved 2, 3;
  reserved "email", "phone";
  int32 id = 1;
  string name = 4;
}
----

Reserving fields is critical for maintaining forward and backward compatibility because it prevents accidental reuse of tag numbers or field names.


[id="registry-compatibility-openapi_{context}"]
== OpenAPI compatibility

[role="_abstract"]
{registry} checks OpenAPI specification compatibility by comparing operations and schemas between versions.
Changes that remove or restrict existing API contracts are considered incompatible.

=== Backward compatible changes for OpenAPI

You can make these changes to an OpenAPI specification and maintain backward compatibility.
Existing API clients continue to operate without modification.

.OpenAPI backward compatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Add a new path or endpoint
|Existing clients are not affected by new endpoints that they do not call.

|Add a new optional query parameter
|Existing clients can continue making requests without the new parameter.

|Add a new optional response header
|Existing clients can ignore the new header in responses.

|Relax request body schema constraints
|Existing requests that the old schema validated remain valid when you relax the constraints.

|Add a new response status code
|Existing clients typically handle unexpected status codes gracefully.
|===

=== Backward incompatible changes for OpenAPI

If you make these changes, you break backward compatibility.
Existing API clients might receive errors or unexpected responses.

.OpenAPI backward incompatible changes
[%header,cols="2,3"]
|===
|Change
|Description

|Remove a path or endpoint
|Existing clients that depend on the removed endpoint receive errors.

|Remove an operation from a path
|Existing clients that use the removed HTTP method (for example, `GET` or `POST`) receive errors.

|Add a new required request parameter
|Existing client requests that do not include the new parameter are rejected.

|Tighten request body schema constraints
|Existing client requests that the old schema validated might be rejected under the new schema.

|Change a response body schema incompatibly
|Existing clients that parse the response according to the old schema might fail.

|Remove a response status code
|Existing clients that handle the removed status code might not handle the replacement correctly.
|===
