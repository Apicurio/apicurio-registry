= Deploying {registry} for high availability
:_mod-docs-content-type: ASSEMBLY

[role="_abstract"]
You can deploy {registry} for high availability in a single {kubernetes} cluster by running multiple stateless replicas with a highly available storage backend.

[id="ha-architecture-overview_{context}"]
== High availability architecture overview
:_mod-docs-content-type: CONCEPT

[role="_abstract"]
A highly available {registry} deployment runs multiple stateless replicas distributed across failure domains, backed by a highly available storage layer. This architecture eliminates single points of failure for both the application tier and the data tier.

A highly available {registry} deployment consists of the following components:

.High availability components
[cols="1,3", options="header"]
|===
|Component |Description

|Application (backend) pods
|Multiple stateless replicas that process REST API requests. These pods scale horizontally and are distributed across cluster nodes for fault tolerance.

|UI pods
|Multiple stateless replicas serving the web console (static files). Typically fewer replicas are needed compared to the backend.

|Storage layer
|The stateful component requiring HA configuration. The strategy depends on the storage type: SQL database, KafkaSQL, GitOps, or KubernetesOps.
|===

The application and UI components are stateless and scale horizontally. High availability is achieved by running multiple replicas distributed across failure domains (availability zones) and implementing a highly available storage layer.

[id="ha-application-scaling_{context}"]
== Application component high availability
:_mod-docs-content-type: CONCEPT

[role="_abstract"]
The {registry} backend and UI components are stateless and scale horizontally. You can configure multiple replicas, distribute them across nodes and availability zones, set resource limits, and define disruption budgets to maintain availability during planned operations.

**Multiple replica configuration**

Configure the number of replicas for each component in the `ApicurioRegistry3` custom resource:

[source,yaml]
----
apiVersion: registry.apicur.io/v1
kind: ApicurioRegistry3
metadata:
  name: example-registry-ha
spec:
  app:
    replicas: 3
    storage:
      type: postgresql
      sql:
        dataSource:
          url: jdbc:postgresql://postgresql-ha.my-project.svc:5432/registry
          username: registry_user
          password:
            name: postgresql-credentials
            key: password
    ingress:
      host: registry.example.com
  ui:
    replicas: 2
    ingress:
      host: registry-ui.example.com
----

IMPORTANT: Running multiple replicas requires a production-ready storage backend (PostgreSQL, MySQL, or KafkaSQL
with persistent Kafka). Do not use the default embedded H2 configuration with multiple replicas.

**Pod distribution across nodes**

To ensure high availability, distribute {registry} pods across different nodes and availability zones:

[source,yaml]
----
spec:
  app:
    replicas: 3
    podTemplateSpec:
      spec:
        affinity:
          podAntiAffinity:
            preferredDuringSchedulingIgnoredDuringExecution:
              - weight: 100
                podAffinityTerm:
                  labelSelector:
                    matchLabels:
                      app: example-registry
                      app.kubernetes.io/component: app
                      app.kubernetes.io/part-of: apicurio-registry
                  topologyKey: kubernetes.io/hostname
              - weight: 50
                podAffinityTerm:
                  labelSelector:
                    matchLabels:
                      app: example-registry
                      app.kubernetes.io/component: app
                      app.kubernetes.io/part-of: apicurio-registry
                  topologyKey: topology.kubernetes.io/zone
----

This configuration spreads pods across different nodes (hostname) and availability zones when possible.

**Resource requests and limits**

Set appropriate resource requests and limits to ensure pod scheduling and prevent resource contention:

[source,yaml]
----
spec:
  app:
    podTemplateSpec:
      spec:
        containers:
          - name: apicurio-registry-app
            resources:
              requests:
                memory: "512Mi"
                cpu: "500m"
              limits:
                memory: "1Gi"
                cpu: "1000m"
----

**PodDisruptionBudget configuration**

Ensure minimum availability during voluntary disruptions such as node drains or cluster upgrades. The operator
creates a PodDisruptionBudget with `maxUnavailable: 1` by default, but you can customize it by creating your own:

[source,yaml]
----
spec:
  app:
    replicas: 3
    podDisruptionBudget:
      enabled: false
----

[source,yaml]
----
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
    name: example-registry-app-poddisruptionbudget
spec:
    minAvailable: 1
    selector:
      matchLabels:
        app: example-registry
        app.kubernetes.io/component: app
        app.kubernetes.io/part-of: apicurio-registry
----

[id="ha-sql-storage_{context}"]
== Configuring SQL database storage for high availability
:_mod-docs-content-type: PROCEDURE

[role="_abstract"]
When you use PostgreSQL or MySQL storage, high availability depends on your database configuration. You must configure the database with replication and automatic failover before running multiple {registry} replicas.

NOTE: MySQL storage is not yet supported by the operator and requires manual configuration using environment variables.

**Database high availability options**

.Database high availability strategies
[cols="2,3", options="header"]
|===
|Strategy |Description

|PostgreSQL with streaming replication
|Primary-replica configuration with automatic failover using tools such as Patroni, CloudNativePG, or Crunchy PostgreSQL Operator

|PostgreSQL with synchronous replication
|Ensures zero data loss but may impact performance

|MySQL with Group Replication
|Multi-primary or single-primary mode with automatic failover

|Managed database services
|Cloud provider managed databases (RDS, Cloud SQL, Azure Database) with built-in HA
|===

**Connection pool configuration for failover**

Configure the Agroal connection pool to handle database failover gracefully. Add these environment variables
to the `app` component:

[source,yaml]
----
spec:
  app:
    env:
      # Connection pool sizing
      - name: APICURIO_DATASOURCE_JDBC_INITIAL-SIZE
        value: "10"
      - name: APICURIO_DATASOURCE_JDBC_MIN-SIZE
        value: "10"
      - name: APICURIO_DATASOURCE_JDBC_MAX-SIZE
        value: "50"

      # Connection acquisition timeout (5 seconds)
      - name: QUARKUS_DATASOURCE_JDBC_ACQUISITION-TIMEOUT
        value: "5S"

      # Background validation to detect stale connections (every 2 minutes)
      - name: QUARKUS_DATASOURCE_JDBC_BACKGROUND-VALIDATION-INTERVAL
        value: "2M"

      # Foreground validation before use (every 1 minute)
      - name: QUARKUS_DATASOURCE_JDBC_FOREGROUND-VALIDATION-INTERVAL
        value: "1M"

      # Maximum connection lifetime (30 minutes)
      - name: QUARKUS_DATASOURCE_JDBC_MAX-LIFETIME
        value: "30M"
----

These settings ensure that:

* Connections are validated regularly to detect database failovers
* Stale connections are removed and recreated
* Connection acquisition times out rather than hanging indefinitely

**Example: PostgreSQL with CloudNativePG**

When you use the CloudNativePG operator for PostgreSQL HA:

[source,yaml]
----
apiVersion: postgresql.cnpg.io/v1
kind: Cluster
metadata:
  name: registry-db-cluster
spec:
  instances: 3
  storage:
    size: 20Gi
    storageClass: standard
  postgresql:
    parameters:
      max_connections: "200"
  backup:
    barmanObjectStore:
      # Configure backup storage
----

Then reference the cluster in your `ApicurioRegistry3` CR:

[source,yaml]
----
spec:
  app:
    storage:
      type: postgresql
      sql:
        dataSource:
          url: jdbc:postgresql://registry-db-cluster-rw:5432/app
          username: app
          password:
            name: registry-db-cluster-app
            key: password
----

[id="ha-kafkasql-storage_{context}"]
== KafkaSQL storage high availability
:_mod-docs-content-type: CONCEPT

[role="_abstract"]
When you use KafkaSQL storage, high availability depends on the Kafka cluster configuration. Each {registry} replica independently consumes all messages from the Kafka journal topic, which means you can add or restart replicas without affecting others.

**Kafka cluster high availability**

Configure your Kafka cluster for high availability:

[source,yaml]
----
apiVersion: kafka.strimzi.io/v1beta2
kind: Kafka
metadata:
  name: registry-kafka
spec:
  kafka:
    version: 3.5.0
    replicas: 3
    config:
      # Replication settings for HA
      offsets.topic.replication.factor: 3
      transaction.state.log.replication.factor: 3
      transaction.state.log.min.isr: 2
      default.replication.factor: 3
      min.insync.replicas: 2
    storage:
      type: persistent-claim
      size: 100Gi
  zookeeper:
    replicas: 3
    storage:
      type: persistent-claim
      size: 10Gi
----

Key configuration for HA:

.Key Kafka configuration for high availability
[cols="2,3", options="header"]
|===
|Configuration |Purpose

|`replicas: 3`
|Provides fault tolerance for broker failures.

|`default.replication.factor: 3`
|Ensures each partition has three copies.

|`min.insync.replicas: 2`
|Requires at least two replicas to acknowledge writes before returning success.
|===

**KafkaSQL topic configuration**

{registry} uses three Kafka topics for various purposes:

Journal topic (`kafkasql-journal`)::
Stores all changes to the registry data. This is the most important topic for data durability.

Snapshots topic (`kafkasql-snapshots`)::
Stores periodic snapshots of the registry state for faster startup. Configure with similar replication settings as the journal topic.

Events topic (`registry-events`)::
Stores events for the Kafka-based eventing feature. Configuration depends on your event processing needs. {registry} sends messages to only 1 partition of this topic.

Configure the Kafka topics used by {registry} with appropriate replication:

[source,yaml]
----
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaTopic
metadata:
  name: kafkasql-journal
  labels:
    strimzi.io/cluster: registry-kafka
spec:
  partitions: 3
  replicas: 3
  config:
    cleanup.policy: delete
    min.insync.replicas: 2
    retention.ms: -1  # Infinite retention
    retention.bytes: -1  # Infinite retention
----

WARNING: The journal topic and the snapshots topic must use `cleanup.policy: delete` with infinite retention (`retention.ms: -1` and `retention.bytes: -1`) to prevent accidental data loss. As of the latest version, {registry} will check these settings on startup and refuse to start if they are not configured correctly.

{registry} automatically creates the journal, snapshots, and events topics on startup by default, if they do not exist. While this is not recommended in a high-availability production scenario, if you want the topics to be created automatically, the following environment variables provide the equivalent configuration:

[source,bash]
----
APICURIO_KAFKASQL_TOPIC_PARTITIONS=3
APICURIO_KAFKASQL_TOPIC_REPLICATION_FACTOR=3
APICURIO_KAFKASQL_TOPIC_MIN_INSYNC_REPLICAS=2
APICURIO_KAFKASQL_SNAPSHOTS_TOPIC_PARTITIONS=3
APICURIO_KAFKASQL_SNAPSHOTS_TOPIC_REPLICATION_FACTOR=3
APICURIO_KAFKASQL_SNAPSHOTS_TOPIC_MIN_INSYNC_REPLICAS=2
----

**Consumer behavior with multiple replicas**

When running multiple {registry} replicas with KafkaSQL storage:

* Each replica uses a unique consumer group ID (automatically generated using UUID)
* Each replica independently consumes all messages from the journal topic
* There is no consumer group rebalancing between replicas
* All replicas build the same local state from the Kafka topic ({registry} replays data into an embedded H2 database)

This design ensures that:

* New replicas can be added without affecting existing replicas
* Pod restarts only affect the restarting pod, not others
* Each replica maintains a consistent view of the data

WARNING: Do not configure a fixed Kafka consumer group ID (for example, by setting
`APICURIO_KAFKASQL_CONSUMER_GROUP_ID`). KafkaSQL uses an embedded H2 database that is empty on every
restart, so the full journal topic must be replayed from the beginning each time. A fixed consumer
group ID causes Kafka to resume from previously committed offsets, skipping the journal replay and
resulting in the registry appearing empty after a restart. If you need a recognizable consumer group
name for monitoring, use the `APICURIO_KAFKASQL_CONSUMER_GROUP_PREFIX` environment variable instead,
which sets a prefix while still generating a unique group ID on each startup.

[id="ha-monitoring_{context}"]
== High availability monitoring
:_mod-docs-content-type: CONCEPT

[role="_abstract"]
{registry} exposes Prometheus metrics that enable you to monitor application health and performance in high availability deployments. You can configure the Prometheus Operator to scrape metrics and set up alerts for common failure conditions.

**Metrics endpoint**

Metrics are enabled by default. Access the metrics endpoint at `/metrics` on the management port (default 9000).

**Key metrics to monitor**

Monitor these metrics for HA deployments:

.Key metrics for high availability deployments
[cols="2,1,3", options="header"]
|===
|Metric |Category |Description

|`http_server_requests_seconds`
|REST API
|Request latency

|`http_server_active_requests`
|REST API
|Concurrent requests

|`http_server_requests_total`
|REST API
|Total request count

|`apicurio_storage_operation_seconds`
|Storage
|Storage operation latency

|`apicurio_storage_concurrent_operations`
|Storage
|Concurrent storage operations

|`apicurio_storage_operation_total`
|Storage
|Total storage operations

|`jvm_memory_used_bytes`
|JVM
|Memory usage

|`jvm_gc_pause_seconds`
|JVM
|Garbage collection pauses

|`/health/ready`
|Health check
|Readiness probe endpoint (management port 9000)

|`/health/live`
|Health check
|Liveness probe endpoint (management port 9000)
|===

**ServiceMonitor configuration for Prometheus Operator**

If using the Prometheus Operator, create a ServiceMonitor to scrape metrics:

[source,yaml]
----
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: apicurio-registry-metrics
  labels:
    app: apicurio-registry
spec:
  selector:
    matchLabels:
      app: apicurio-registry
  endpoints:
    - port: management
      path: /metrics
      interval: 30s
----

**Alerting recommendations**

Configure alerts for:

* Pod restarts or crash loops
* High error rates (5xx responses)
* Storage operation timeouts
* Database connection pool exhaustion
* Kafka consumer lag (for KafkaSQL storage)

[id="ha-rolling-updates_{context}"]
== Rolling updates
:_mod-docs-content-type: CONCEPT

[role="_abstract"]
Rolling updates allow you to update {registry} to a new version with minimal downtime by replacing pods one at a time, ensuring that readiness checks pass before proceeding to the next pod.

**Rolling update strategy**

The operator performs rolling updates automatically when you update the `ApicurioRegistry3` CR. The default
strategy ensures:

* Pods are updated one at a time
* Each new pod must pass readiness checks before the next pod is updated
* PodDisruptionBudget prevents too many pods being unavailable

**{registry} version updates**

{registry} and {operator} are versioned together. To update {registry} to a new version, update {operator}, which will automatically update the application.

For production environments, we strongly recommend using Operator Lifecycle Manager (OLM) with manual install plan confirmation enabled. This prevents automatic updates from occurring immediately when a new version becomes available, giving you time to review release notes and prepare for the update.

If you need to update the application without updating {operator} (*not recommended* except as a workaround for critical issues), you can override the application image in your pod template:

[source,yaml,subs="+attributes"]
----
spec:
  app:
    podTemplateSpec:
      spec:
        containers:
          - name: apicurio-registry-app
            image: quay.io/apicurio/apicurio-registry:{registry-release}
----

**Safe update practices**

Follow these practices for safe updates:

.Safe update practices
[cols="2,3", options="header"]
|===
|Practice |Details

|Test in non-production first
|Validate the new version in a test environment before rolling out to production.

|Monitor during rollout
|Watch metrics and logs during the update.

|Maintain minimum replicas
|Keep at least 2 replicas to ensure availability during updates.

|Review release notes
|Check for breaking changes or migration steps before upgrading.
|===

IMPORTANT: For patch releases (e.g., 3.1.0 to 3.1.1), rolling updates typically complete without issues. For major or minor version updates, always review the migration guide.

**Management interface upgrade (port 9000)**

Starting with {registry} 3.x, health check and metrics endpoints are served on a dedicated management port (9000) instead of the main application port (8080). This is a *breaking change* that requires updates to your deployment configuration when upgrading from earlier versions.

The following components require configuration updates:

Kubernetes health probes::
Update liveness and readiness probe definitions to target port `9000`. The endpoint paths remain the same.
+
[source,yaml]
----
# Before (old configuration)
livenessProbe:
  httpGet:
    path: /health/live
    port: 8080

# After (new configuration)
livenessProbe:
  httpGet:
    path: /health/live
    port: 9000

# After (with TLS enabled)
livenessProbe:
  httpGet:
    path: /health/live
    port: 9000
    scheme: HTTPS
----

Prometheus scrape targets::
Update your Prometheus configuration or `ServiceMonitor` resources to scrape metrics from port `9000` at path `/metrics`.

Network policies::
If you use Kubernetes `NetworkPolicy` resources, ensure that ingress traffic on port `9000` is allowed from your monitoring infrastructure and the kubelet (for health probes).

TLS configuration::
The management interface inherits the default TLS configuration from the Quarkus TLS registry. When TLS is enabled, health probes must use `scheme: HTTPS`. The {operator} handles this automatically.

{operator} managed deployments::
The {operator} automatically configures the correct probe ports. No manual changes are required.

[id="ha-backup-restore_{context}"]
== Backup and restore
:_mod-docs-content-type: CONCEPT

[role="_abstract"]
Regular backups protect your {registry} data from accidental loss and enable disaster recovery. The backup strategy depends on your storage type: SQL database (PostgreSQL or MySQL) or KafkaSQL storage.

**SQL database backup**

For SQL storage (PostgreSQL or MySQL), backup strategies depend on your database setup:

**PostgreSQL backup options**

Use `pg_dump` for logical backups:

[source,bash]
----
pg_dump -h postgresql-host -U registry_user -d registry > registry-backup.sql
----

For point-in-time recovery, use continuous archiving with WAL.

If you use the CloudNativePG or Crunchy PostgreSQL Operator, configure operator-managed backups:

[source,yaml]
----
spec:
  backup:
    barmanObjectStore:
      destinationPath: s3://my-backups/registry-db
      s3Credentials:
        accessKeyId:
          name: backup-credentials
          key: ACCESS_KEY_ID
        secretAccessKey:
          name: backup-credentials
          key: ACCESS_SECRET_KEY
      wal:
        compression: gzip
    retentionPolicy: "30d"
----

**MySQL backup options**

Use `mysqldump` for logical backups:

[source,bash]
----
mysqldump -h mysql-host -u registry_user -p registry > registry-backup.sql
----

For physical backups, use tools such as Percona XtraBackup or MySQL Enterprise Backup.

**SQL backup restoration**

To restore from a logical backup:

[source,bash]
----
# PostgreSQL
psql -h postgresql-host -U registry_user -d registry < registry-backup.sql

# MySQL
mysql -h mysql-host -u registry_user -p registry < registry-backup.sql
----

**KafkaSQL storage backup**

For KafkaSQL storage, back up the Kafka journal topic and snapshots topic.

**Kafka topic backup**

Use Kafka's MirrorMaker 2 or {kafka-streams} Mirror Maker for topic replication (recommended):

[source,yaml]
----
apiVersion: kafka.strimzi.io/v1beta2
kind: KafkaMirrorMaker2
metadata:
  name: registry-backup-mirror
spec:
  version: 3.5.0
  replicas: 1
  connectCluster: "backup-cluster"
  clusters:
    - alias: "source-cluster"
      bootstrapServers: registry-kafka-bootstrap:9092
    - alias: "backup-cluster"
      bootstrapServers: backup-kafka-bootstrap:9092
  mirrors:
    - sourceCluster: "source-cluster"
      targetCluster: "backup-cluster"
      topicsPattern: "kafkasql-.*"
      groupsPattern: ".*"
----

Alternatively, export and import topics using Kafka's console consumer or CLI tools like `kcat`/`kafkacat`. Since the topics might contain binary data, which might be affected by text file encoding, make sure they are exported correctly when using CLI tools. See our xref:getting-started/guide-exporting-registry-kafka-topic-data.adoc[] guide for more details.

**Backup and restore verification**

Regularly test your backup and restore procedures:

. Create a test {registry} deployment in a separate namespace
. Restore from backup to the test deployment
. Verify that all artifacts and metadata are present
. Test API functionality to ensure data integrity
