How It Works: The Projection Model

ODCS contracts are stored as ODCS_CONTRACT artifacts and their contents are projected onto schema artifacts for enforcement.

ODCS YAML
submitted by developer
→
Parser
validates structure
→
Projection Engine
maps to labels/rules/tags
→
Schema Artifact
enforcement happens here
ODCS SectionProjected To
info.status, team.*contract.{contractId}.* artifact labels
quality.accuracy rulesCEL contract rules (odcs:{contractId}:*)
schemas[].fields[].piifield-tag.{contractId}:path|tag version labels
serviceLevel.*contract.{contractId}.sla.* labels (stored, not enforced)

Zero new storage tables for core contracts — reuses existing label and rule infrastructure. Works across SQL, KafkaSQL, GitOps, KubernetesOps.

Speaker notes: The key insight: we don't create a new storage model. We project the contract onto the schema artifact it governs. Quality rules become CEL rules. Metadata becomes labels. PII markers become version tags. Enforcement happens through the existing rule engine. Everything is namespaced by contractId so multiple contracts can coexist. The ODCS YAML is also stored as its own artifact for versioning and export. Let me show you this in action.

▶ DEMO Step 2: Submit ODCS Contract

ODCS v3.1 contract

apiVersion: v3.1.0
kind: DataContract
id: orders-contract
info:
  title: Orders Data Contract
  version: 1.0.0
  status: active
  dataClassification: confidential
team:
  name: orders-team
  domain: commerce
schemas:
  - name: OrderEvent
    type: avro
    location: odcs-example/OrderEvent:latest
    fields:
      customerEmail:
        pii: true
        tags: [PII, EMAIL]
quality:
  accuracy:
    - name: positive-amount
      expression: totalAmount > 0
      threshold: 1.0

Expected output

▶ DEMO Steps 6-7: CEL Rule Engine

How ODCS accuracy rules become CEL

# ODCS YAML:
quality:
  accuracy:
    - name: positive-amount
      expression: totalAmount > 0
      threshold: 1.0     # → onFailure: ERROR

# Projected as:
odcs:orders-contract:positive-amount
  type: CEL
  expr: totalAmount > 0
  onFailure: ERROR

Threshold mapping

  • 1.0 → ERROR (reject)
  • 0.95-0.99 → DLQ (dead letter queue)
  • <0.95 → NONE (log only)

Expected output

▶ Step 6: Verify ODCS-Projected Rules
ODCS rule: odcs:orders-contract:positive-amount
[CEL] → totalAmount > 0
✓ Added JSONata migration rule

▶ Step 7: Execute CEL Rules
✓ Valid (totalAmount=99.99): {"passed":true}
✗ Invalid (totalAmount=-5.0): {"passed":false,
"violations":["positive-amount"]}

Full CEL spec: has(), size(), contains(), matches(), ternary, parenthesized grouping.

Speaker notes: Press Enter for step 6 (verify rules + add migration rule), then step 7 (execute). The demo sends a valid record (passes) then an invalid one (totalAmount=-5, rejected). This is the same engine that runs in SerDes later.

▶ DEMO Steps 11-14: Kafka SerDes Integration

Config-only integration

# Just two properties — that's it:
apicurio.registry.serde
  .contract-rules.enabled=true
apicurio.registry.serde
  .contract-rules.fail-on-error=true

What happens

  • Step 11: Start embedded Kafka (Testcontainers)
  • Step 12: Produce valid message → rules pass → sent
  • Step 13: Produce invalid message → rules reject → blocked
  • Step 14: Consume valid message → read-side rules pass

DLQ pattern

Set fail-on-error=false → violations logged but message serialized. App routes to DLQ.

Speaker notes: Two modules. The app module has everything that needs storage: ODCS parser/projectors, tag extractors, governance, audit. The contracts-rules module is storage-free — it contains the rule execution engine, CEL, and JSONata. This module is reused by Kafka SerDes without pulling in the entire app.

Storage: Atomic Label Merge

SQL Storage

-- In one database transaction:
DELETE FROM artifact_labels
  WHERE groupId=? AND artifactId=?
  AND labelKey LIKE 'contract.orders.%';

INSERT INTO artifact_labels ...
  ('contract.orders.status', 'STABLE'),
  ('contract.orders.owner.team', 'orders');

-- Rebuild JSON column
SELECT → UPDATE artifacts SET labels=?

KafkaSQL Storage

MergeArtifactLabels5Message {
  groupId, artifactId,
  prefix: "contract.orders.",
  labels: {"status":"STABLE", ...}
}
  │ published to Kafka journal
  ▼ consumed by ALL replicas
  ▼ each applies atomic merge
  → strong consistency via ordering
  • Different contracts → different prefixes → no conflict
  • Same contract → idempotent overwrite
  • ReadOnly decorator guards all write methods
  • Only 1 new table: contract_audit_log (everything else is labels + rules)
Speaker notes: Two storage variants, same consistency. SQL: ACID transaction. KafkaSQL: merge instruction goes through the Kafka journal, all replicas apply in order. Same pattern as every other KafkaSQL write.
